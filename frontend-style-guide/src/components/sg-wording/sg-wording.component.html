Wording

General Rules

  • Use familiar domain language that users already know from real life
  • Use short, direct sentences and avoid technical system terms in the interface
  • Use the same word for the same concept everywhere, for example Product, User, Save and Cancel
  • Use action labels that describe the result of the action, such as Save, Delete or Add product
  • Only use abbreviations when they are already common to the target users

Navigation Labels

  • Navigation labels must use familiar domain language
  • Labels must be short, scannable and noun-based, for example Products or Users
  • Do not use implementation terms such as Dashboard route, Entity list or CRUD page
  • Icon-only navigation actions require an aria-label that names the action or destination

Feedback & Status Messages

  • Notifications must be short and explain the result or next action
  • Feedback messages must make the current system status clear after user actions
  • Status labels must use plain words users already understand, such as Success, Warning, Error and Information
  • Success messages confirm what happened, for example Product saved successfully
  • Error messages explain what is wrong and how to correct it
  • API errors must be mapped to user-friendly messages before they are displayed

Form Wording

  • Field labels must describe the information users should enter, for example Product name or Price
  • Placeholder text may give an example, but it must not replace a visible label
  • Helper text explains constraints before users make mistakes
  • Validation messages appear close to the field and use the same term as the field label
  • Error messages explain the problem and the next fix in plain language
