Technical Guidelines

Framework & Libraries

  • PrimeNG (UI components)
  • PrimeFlex (layout & spacing)
  • Angular v21.2.9
  • PrimeNG v21.1.6

Approved Library Usage

  • Angular standalone components are the standard implementation model
  • PrimeNG is the approved component library for buttons, tables, dialogs, selects and form controls
  • PrimeIcons is the only approved icon library
  • PrimeFlex may be used for layout utilities, but spacing must still follow the documented 8pt scale
  • Custom components are only allowed when PrimeNG does not provide the required behaviour

Usage Principles

  • Prefer PrimeNG components over custom implementations
  • Extend existing components instead of overriding
  • Maintain separation of concerns

Coding Standards

  • Modular structure:
    • /components
    • /pages
    • /services
    • /assets
  • Separation of logic, layout, and styling
  • No inline styles unless necessary
  • Use CSS custom properties and documented styleClass values for colours, spacing, typography and component states
  • Do not create new component-specific style classes for patterns already documented in the guide
  • Keep templates declarative; move calculations, mapping and API orchestration into TypeScript services or page logic

API Integration

  • UI is independent of API logic
  • Service layer handles API communication
  • Components call services or facades, never backend endpoints directly
  • Services return typed UI-ready data or typed state objects
  • Standard states:
    • Loading
    • Error
    • Empty

API Call Rules

  • Components must not call HTTP APIs directly
  • API calls must be wrapped in Angular services in /services
  • Services expose typed methods such as getProducts(), createProduct(product), updateProduct(product) and deleteProduct(id)
  • Components receive loading, error, empty and success states from services or a page-level facade
  • API errors must be mapped to messages that follow the wording rules before they are displayed
  • State changes must be surfaced through visible feedback so users understand what is happening
  • Destructive API calls must require confirmation before the request is sent

Frontend / Backend Connection

  • Backend DTOs must be mapped to UI models before rendering
  • Backend field names must not leak into templates when they differ from user-facing terminology
  • Forms submit validated UI models to services, not raw template values
  • Tables consume display-ready data and must not transform API payloads in templates
  • Components display loading, empty, success and error states provided by services or page facades
  • Notifications and dialogs are handled by shared services so feedback is consistent across pages
  • Shared services should keep feedback, errors and recovery actions consistent across similar workflows

Versioning

  • Current style guide version: 1.1.0
  • Application implementations must state which style guide version they follow
  • Semantic versioning:
    • 1.0.0 initial release
    • 1.1.0 feature updates
    • 1.1.1 bug fixes
