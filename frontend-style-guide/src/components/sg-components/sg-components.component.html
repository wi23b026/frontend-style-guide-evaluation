Components

General Component Principles

  • Only approved UI components may be used
  • Behaviour, usage and appearance must be standardised
  • Component size and interaction patterns are predefined
  • Visual examples define correct usage
  • Consistent handling across all UI elements is required
  • System feedback must be visible through defined hover, focus, active, disabled, loading, validation and status states
  • Error handling must use clear messages, accessible state attributes and safe recovery actions
  • Status indicators must combine icon, colour and text so meaning never depends on colour alone
  • Developers should use the documented styleClass values exactly as shown in examples

Approved Component Catalogue

  • Buttons: PrimeNG p-button with sg-button-primary, sg-button-primary-inverted, sg-button-save, sg-button-cancel or sg-button-dark
  • Dropdowns: PrimeNG p-select with sg-select and sg-select-panel
  • Tables: PrimeNG p-table with sg-table
  • Inputs: PrimeNG pInputText with sg-input and validation classes when needed
  • Cards: approved content card pattern with sg-card classes
  • Accordions: PrimeNG p-accordion pattern for expandable content groups
  • Tabs: PrimeNG p-tabs pattern for switching between related views
  • Notifications: shared toast/alert pattern with icon, colour and text
  • Navigation: approved navbar/menu pattern with semantic nav markup
  • Dialogs: PrimeNG p-dialog with sg-dialog
  • Icons: PrimeIcons only
  • Autocomplete: approved custom input plus suggestion dropdown pattern when PrimeNG p-autoComplete is not used

Buttons

Usage

Primary interaction element for actions.

Example

Normal State (Active / Focus / Disabled)
Save State (Active / Focus / Disabled)
Cancel State (Active / Focus / Disabled)

Characteristics

  • PrimeNG p-button only
  • Buttons must be rectangular only; circular or fully rounded icon buttons are not allowed
  • Normal state: white background (#FFFFFF) with dark brown border and text (#372110) for primary actions
  • Normal inverted state: dark brown background (#372110) with white text (#FFFFFF) for high-emphasis primary actions on light surfaces
  • Save state: green background (#6FCF97) with white text for committing changes
  • Cancel state: red background (#E85C5C) with white text for dismissive actions
  • Icon-to-text gap: 0.5rem
  • Padding left/right: 1rem
  • Padding bottom per state group: 1rem
  • Margin between buttons: 1rem
  • Margin between button state groups: 1.5rem
  • 0.5rem border radius for all buttons
  • Minimum height: 2.75rem
  • Size: text buttons use content width with 1rem horizontal padding; icon-only buttons use 2.75rem by 2.75rem
  • Responsive behaviour: button groups wrap on medium viewports and stack full-width on narrow form layouts

Cards

Usage

Compact content container for grouped summary information.

Example

Taiyaki Overview
3 Total products

Characteristics

  • Use cards only for grouped summary or repeated item content
  • Do not place page sections inside decorative cards
  • Border radius: 0.5rem maximum
  • Use visible heading, content and optional action areas
  • Cards must not contain other cards
  • Size: cards fill their parent container up to the page content width and use at least 1.5rem internal padding
  • Responsive behaviour: repeated cards use a responsive grid and collapse to one column on narrow viewports

Tables

Usage

Structured display for product data, records, and comparable information.

Example

Name Price Category {{ product.name }} {{ product.price }} {{ product.category }}

Characteristics

  • Header background: sand light (#FFEDE0) with primary-light bottom border
  • Row background: odd rows white, even rows light orange (#fff8f2)
  • Hover background: white rows change to #fff8f2; even rows keep #fff8f2
  • Standard table actions appear in the final column
  • Empty tables must show a visible empty state message
  • Cell padding: 1rem
  • Border radius: 0.5rem on the table wrapper
  • Size: table columns use content-based widths with a minimum table width of 40rem when several columns are shown
  • Filtering: use PrimeNG p-table filtering APIs, such as column filters or global filter fields, instead of custom DOM filtering
  • Sorting and pagination: use PrimeNG p-table inputs and templates when needed
  • Responsive behaviour: table wrapper allows horizontal scrolling for narrow viewports

Accordions

Usage

Expandable groups for secondary information that does not need to be visible all at once.

Example

@if (accordionOpen) {
Ingredients, availability and preparation notes appear inside the expanded panel.
}

Characteristics

  • Use the PrimeNG accordion pattern for implementation
  • Headers must be keyboard reachable and show a visible focus state
  • Use chevron icons to communicate expanded or collapsed state
  • Accordion content should be secondary, not required for completing the main task
  • Use aria-expanded or the equivalent PrimeNG accessibility behaviour
  • Size: accordion headers use a minimum height of 2.75rem and panels use 1rem internal padding
  • Responsive behaviour: accordion width follows the parent container and long header text wraps without hiding the chevron

Tabs

Usage

Switch between related views on the same page without changing navigation context.

Example

{{ tabContent[activeTab] }}

Characteristics

  • Use the PrimeNG tabs pattern for implementation
  • Tabs are for peer-level views, not step-by-step processes
  • Active tab must be visually distinct and announced with aria-selected or equivalent behaviour
  • Tab labels must follow the wording rules
  • Do not use tabs when standard page navigation is clearer
  • Size: tab controls use a minimum height of 2.75rem and at least 1rem horizontal padding
  • Responsive behaviour: tab lists scroll horizontally or wrap when labels do not fit; tab panels keep full parent width

Dropdowns

Usage

Selection control for choosing one option from a predefined list.

Example

Characteristics

  • Component: PrimeNG p-select only
  • Dropdowns must use the same closed control and overlay pattern everywhere
  • Closed state: white background, neutral border, dark brown text
  • Open/Focus state: accent red border with visible focus ring
  • Overlay background: white with 0.5rem border radius and visible shadow
  • Option hover/focus: sand-light background
  • Selected option: primary-light background with dark brown text
  • Minimum height: 2.75rem
  • Size: closed dropdowns fill their form column and should not be narrower than 12rem
  • Responsive behaviour: dropdown overlays align to the input width and remain usable on narrow viewports

Autocomplete

Usage

Text input with visible suggestions for known values such as product names or searchable labels.

Example

Red Bean Taiyaki
Custard Taiyaki
Matcha Taiyaki
Suggestions appear below the input and can scroll when more than five options are available.

Characteristics

  • Suggestions are shown in a dropdown directly below the input
  • Show suggestions while the input is focused or while the user types
  • Limit the visible list to five suggestions, then make the panel scrollable
  • Keyboard support must include ArrowDown, ArrowUp, Enter and Escape
  • Empty suggestion lists should not open an empty panel
  • The input must keep a visible label and use aria-describedby for helper or error text
  • Size: autocomplete inputs follow input field sizing; suggestion panels match the input width with a maximum visible height of five options
  • Responsive behaviour: suggestion panels stay attached below the input and scroll internally instead of extending past the viewport

Input Fields

Usage

Text entry for forms, search, authentication, and editable user input.

Example

Please enter a valid value.
Input accepted.

Characteristics

  • Default state: neutral border and placeholder text for empty or editable input
  • Error state: red border, aria-invalid attribute, and a clear validation message
  • Success state: green border and confirmation message after valid input
  • Used for forms, search, authentication, and editable text fields
  • Clear focus, validation, and helper messaging are required
  • Size: inputs use a minimum height of 2.75rem, fill their form column and should not be narrower than 12rem
  • Responsive behaviour: multi-column forms collapse to fewer columns and then one column on narrow viewports

Notifications / Alerts / Toasts

Usage

Temporary or persistent feedback after system actions.

Example

Product saved successfully.
Product could not be deleted. Try again.

Characteristics

  • Use shared notification services so messages behave consistently
  • Success, warning, error and information states must combine icon, colour and text
  • Use role="status" for passive updates and role="alert" for urgent errors
  • Notification text must follow the wording rules
  • Dismiss controls must have accessible labels when shown
  • Size: notifications use content height, 1rem padding and a maximum width of 25rem
  • Responsive behaviour: notifications fit within the viewport width and stack vertically with consistent spacing

Navigation / Menus / Navbar

Usage

Global and local navigation that helps users understand where they are and where they can go.

Example

TAIYAKI SHOP Products Users

Characteristics

  • Use semantic nav landmarks with accessible labels
  • Global navigation appears at the top of every application page
  • Navigation labels must follow the wording rules
  • Active or focused items must be visually clear
  • Icon-only navigation actions require aria-label
  • Size: global navigation uses a minimum height of 5rem; nav links and icon actions use a minimum height of 2.75rem
  • Responsive behaviour: brand remains first, navigation actions align to the right on desktop and wrap or stack below the brand on narrow viewports

Dialogs / Modals

Usage

Temporary focused surfaces for confirmations, short forms, and important contextual content.

Example

Dialog Content

This is an example dialog/modal component.

Characteristics

  • Focus trap required
  • Keyboard (ESC) support required
  • Clear structure (header, content, actions)
  • Size: dialogs use a maximum width of 32rem for confirmations and short forms, with 1.5rem content padding
  • Responsive behaviour: dialogs shrink to fit narrow viewports with visible margins and scroll internally when content is taller than the viewport

Error Handling & Validation

Usage

Validation prevents avoidable mistakes and explains how users can fix invalid input.

Example

Price must be 0 or higher.
Product name accepted.

Characteristics

  • Required actions should be disabled until required input is present
  • Error messages must follow the wording rules
  • Fields with errors must use aria-invalid="true" and connect messages with aria-describedby
  • Validation should happen close to the field that needs attention
  • Users must be able to cancel, close, or safely recover from interrupted processes
  • Size: validation messages use small text and sit directly below the related control without changing the control height
  • Responsive behaviour: validation messages wrap inside the field column and remain visible when forms collapse to one column

Status Indicators

Usage

Status indicators communicate system or item state without relying on colour alone.

Example

Success
Warning
Error
Information

Characteristics

  • Use one PrimeIcon, one approved colour and one visible text label
  • Green means success, orange means warning, red means error, blue means information
  • Do not communicate status through colour alone
  • Status labels must follow the wording rules
  • Size: status indicators use a minimum height of 1.75rem with compact horizontal padding
  • Responsive behaviour: status labels wrap only when necessary and remain paired with their icon

Safe Interaction Patterns

Usage

Safe interaction patterns help users avoid mistakes and leave a process without losing control.

Example

Characteristics

  • Destructive actions must be visually distinct and require confirmation when data will be lost
  • Cancel or close actions must be visible when users are inside a process
  • Disabled actions must only be used when the required conditions are not met
  • Interfaces should show only necessary actions for the current context
  • Size: safe action groups use the standard button sizes and keep at least 1rem between actions
  • Responsive behaviour: action groups keep primary and destructive actions visible, wrapping or stacking without changing action order

Icons

Usage

Visual support for status, feedback and navigation.

Example

Success
Warning
Error
Info

Characteristics

  • Single icon library only (PrimeIcons)
  • House-owned custom icons are reserved for the logo and brand assets only
  • Consistent icon style required
  • Size: standard icons use 1rem inside text flows and 1.25rem to 1.5rem for standalone status use
  • Responsive behaviour: headings wrap naturally without viewport-based font scaling, and icons stay aligned with their text labels
